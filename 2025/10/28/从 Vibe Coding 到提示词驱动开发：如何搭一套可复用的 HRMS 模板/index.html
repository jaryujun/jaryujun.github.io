从 Vibe Coding 到提示词驱动开发：如何搭一套可复用的 HRMS 模板

Jaryujun
2025-10-28
HRMS, Prompt, vibe-coding

最近我在用 AI 搭一个 HRMS 后台模板。刚开始的做法很“vibe coding”：想到哪做到哪，缺什么补什么，前后端、接口、权限、测试全靠上下文临时拼接。这样的方式适合快速试错，但一旦要把项目沉淀成可复用模板，问题就会集中出现。

比如后端一跑起来，常见坑就来了：Flyway 迁移脚本需要数据库权限、springfox 和 Spring Boot 版本不兼容、Swagger 页面被 Spring Security 拦截，甚至匿名白名单在 SecurityConfig 和 JwtAuthenticationFilter 里维护了两份。回头看，这些问题并不全是“代码写错了”，更多是因为项目里哪些东西是固定约束、哪些东西是可变输入，没有在一开始说清楚。

所以我换了一个思路：先抽取 HRMS 项目的不变量，再把可变部分收敛成一份结构化输入，最后把整个流程拆成“需求收集、项目搭建、基础验证”三段式提示词。这样做的目标不是让 AI 一次性生成完美系统，而是让它稳定地生成一个可继续演化的脚手架。

为什么要先抽取不变量

如果想让 AI 生成出来的东西能长期复用，第一步不是写 Prompt，而是先回答一个问题：这个项目里，到底哪些东西不能变？

对这套 HRMS 模板来说，不变量主要有五类：技术栈、目录结构、接口约定、RBAC 核心模型，以及安全边界。只有先把这些讲清楚，AI 才不会一会儿给你生成 React，一会儿又把权限系统改成另一套命名。

不变量清单

  • 前端技术栈与目录结构：Vue 3 + TypeScript + Vite + Element Plus + Tailwind，目录根为 frontend/
  • 后端技术栈与目录结构：Spring Boot 3.2.5 + Java 17 + MyBatis-Plus + Spring Security + JWT，目录根为 backend/
  • 接口返回结构：统一 ApiResponse 与 PageResponse 包裹
  • RBAC 核心表：sys_user、sys_role、sys_menu、sys_user_role、sys_role_menu
  • 核心接口路径：/api/auth/login、/api/user/info、/api/user/list、/api/role/list、/api/v3/system/menus
  • 安全约束：任何 .env 仅提供占位符或 .env.example，禁止真实密钥

这里最重要的并不是“列一份清单”本身，而是把模型的自由度控制在正确范围内。技术选型、接口形状和权限表结构不应该由 AI 即兴发挥；真正应该开放给生成器的，是项目名、端口、数据库占位符、初始角色和菜单树。

用 ScaffoldSpec 收敛变化

把不变量固定下来之后，下一步就是定义一份足够小、但信息完整的输入结构。我把它叫做 ScaffoldSpec。它的作用很简单：让项目差异都收敛在一个 JSON 里，而不是散落在多轮对话里。

ScaffoldSpec 结构

{
  "project": {
    "name": "string",
    "packageName": "string",
    "frontendPort": 3000,
    "backendPort": 9000,
    "accessMode": "frontend | backend"
  },
  "database": {
    "type": "mysql",
    "host": "placeholder",
    "port": 3306,
    "name": "placeholder",
    "username": "placeholder",
    "password": "placeholder"
  },
  "rbac": {
    "roles": [
      { "code": "R_SUPER", "name": "超级管理员", "description": "..." }
    ],
    "menuTree": [
      {
        "name": "System",
        "path": "/system",
        "component": "/index/index",
        "children": []
      }
    ],
    "seedStrategy": "minimal | standard | custom"
  }
}

这个结构有两个好处。

第一，它足够稳定。无论你是做人事系统、组织管理后台，还是一个轻量 RBAC 管理台，只要基础约束一致，都可以复用这份规范。

第二，它足够可控。所有默认值、占位符和初始化数据都能在这里声明清楚，后面的搭建 Prompt 只需要“执行规范”，而不是重新理解需求。

ScaffoldSpec 示例

下面是一份可以直接拿来喂给生成器的示例：

{
  "project": {
    "name": "hrms-template",
    "packageName": "com.hrms",
    "frontendPort": 3006,
    "backendPort": 9000,
    "accessMode": "backend"
  },
  "database": {
    "type": "mysql",
    "host": "DB_HOST",
    "port": 3306,
    "name": "DB_NAME",
    "username": "DB_USERNAME",
    "password": "DB_PASSWORD"
  },
  "rbac": {
    "roles": [
      { "code": "R_SUPER", "name": "超级管理员", "description": "拥有系统全部权限" },
      { "code": "R_ADMIN", "name": "管理员", "description": "拥有系统管理权限" },
      { "code": "R_USER", "name": "普通用户", "description": "拥有系统普通权限" }
    ],
    "menuTree": [
      {
        "name": "System",
        "path": "/system",
        "component": "/index/index",
        "children": [
          { "name": "User", "path": "/system/user", "component": "/system/user" },
          { "name": "Role", "path": "/system/role", "component": "/system/role" },
          { "name": "Menus", "path": "/system/menu", "component": "/system/menu" }
        ]
      }
    ],
    "seedStrategy": "standard"
  }
}

如果你没有额外需求，其实用这份示例就已经能搭出一个最小可运行的 HRMS 模板。文中后面提到的“HRMS 默认角色与菜单树”，也可以直接使用这份示例里的 roles 和 menuTree。

为什么要拆成三段 Prompt

很多人写 Prompt 时喜欢把“需求理解、代码生成、测试验证”塞进同一段话里，表面上看信息更全，实际上更容易失控。因为这三个阶段关注的目标完全不同：

  • 需求收集阶段，要解决的是输入是否完整
  • 搭建阶段，要解决的是输出是否符合既定约束
  • 验证阶段，要解决的是结果是否真的能构建和通过最小测试

把它们拆开以后，每一段 Prompt 的职责会更清晰，失败也更容易定位。

Prompt 1：需求收集

第一段 Prompt 的职责不是生成代码，而是把需求整理成一份完整的 ScaffoldSpec。如果用户没给全，就自动补默认值，并明确告诉他还缺哪些信息需要追问。

你是“HRMS 模板生成器”的需求收集助手。

目标：
1) 输出一份完整的 ScaffoldSpec JSON（结构固定）。
2) 若输入缺失任何字段，按默认值补齐，并在 JSON 下追加“缺失项追问清单”。

规则：
- 默认值：
  project.frontendPort=3006，project.backendPort=9000，project.accessMode="backend"
  database.type="mysql" 且使用占位符（DB_HOST/DB_NAME/DB_USERNAME/DB_PASSWORD）
  rbac.seedStrategy="standard"
- 若未提供 rbac.roles 或 rbac.menuTree，则使用“HRMS 默认角色与菜单树”。
- 输出只包含 JSON + 追问清单，不要附加解释性文本。

输入：
{用户需求或空}

输出示例：
<JSON>
<缺失项追问清单>

这一段的价值在于“先把输入标准化”。到这里为止，AI 还没有开始生成项目，但它已经把后续搭建所需的参数整理好了。

Prompt 2：搭建指令

第二段 Prompt 才真正负责生成项目结构。它强调的重点不是“生成一个后台系统”，而是“严格对齐当前仓库结构与接口约定”。换句话说，它是一个执行器，不是自由创作者。

你是“HRMS 模板生成器”的搭建执行助手。

输入：
- ScaffoldSpec JSON
- 约束：前端目录结构与技术栈对齐 `frontend/`
- 约束：后端目录结构与技术栈对齐 `backend/`
- 约束：API 接口与响应结构必须包含：
  /api/auth/login
  /api/user/info
  /api/user/list
  /api/role/list
  /api/v3/system/menus
  ApiResponse / PageResponse 包裹
- 约束：RBAC 表结构必须包含 sys_user / sys_role / sys_menu / sys_user_role / sys_role_menu
- 安全约束：禁止任何真实密钥，只能生成 .env.example 或占位符

任务：
1) 根据 ScaffoldSpec 生成模板项目结构（frontend + backend）。
2) 配置后端 application.yml、前端环境变量与 API 基础地址。
3) 生成 schema.sql 与 data.sql，角色与菜单树来自 ScaffoldSpec。
4) 确保接口路径与前端调用一致，且后端返回结构与前端类型对齐。
5) 输出“生成摘要 + 关键文件清单 + 注意事项”，不要输出超长源码。

输出：
- 生成摘要
- 关键文件清单
- 注意事项（含敏感信息提醒）

这一段的核心是“把生成约束写死”。例如统一 ApiResponse / PageResponse、固定接口路径、限定 RBAC 表结构、禁止真实密钥，这些都应该明确写进 Prompt，而不是寄希望于模型自己记住。

Prompt 3：验证指令

第三段 Prompt 只做一件事：用最小成本确认这个模板至少是可构建的。这里我特意加了一个限制条件: 不依赖真实数据库。因为模板阶段最容易出现的问题，并不是业务逻辑是否完整，而是项目骨架能不能跑通。

你是“HRMS 模板生成器”的验证助手。

目标：
执行基础验证，确保模板可构建且通过最小测试。

约束：
- 不依赖真实数据库
- 不使用真实密钥

验证步骤：
1) 前端：
   - 进入 frontend 目录
   - 运行 `pnpm lint`
   - 运行 `pnpm build`
2) 后端：
   - 进入 backend 目录
   - 运行 `mvn test`
   - 若无测试用例，确保 contextLoads 通过即可

通过标准：
- 前端构建成功
- 后端测试通过（或上下文加载成功）

输出：
- 每步执行结果
- 失败原因与最短修复建议

把验证从生成里拆出来之后，一个很直接的好处是：当失败发生时，你能立刻知道问题出在“提示词理解错误”“模板结构错误”还是“工程自身无法构建”，排查成本会低很多。

推荐使用顺序

如果你打算真的把这套东西用起来，我建议严格按下面顺序执行：

  1. 先用 Prompt 1，把需求收敛为一份完整的 ScaffoldSpec
  2. 再把 ScaffoldSpec 交给 Prompt 2，生成前后端模板结构
  3. 最后用 Prompt 3 做基础验证，确认前端能构建、后端能通过最小测试

这三个阶段不要合并，也不要跳步。尤其是在模板项目里，先定义输入、再执行搭建、最后做验证，会比“一段话让 AI 全干完”稳定得多。

Test Plan

为了让这套提示词包具备最基本的可验性，我给自己定了一个很朴素的测试方案：

- 使用 Prompt 1 产出示例 ScaffoldSpec。
- 用 Prompt 2 生成项目后，执行 Prompt 3 的基础验证命令。
- 验证通过标准为：前端构建成功、后端测试通过（或能启动到上下文加载）。

这套测试方案不追求覆盖所有业务细节，它只解决一个更关键的问题：这是不是一个“能继续开发”的起点。如果连这个都保证不了，后面讨论功能扩展其实没有意义。

这套方法真正解决了什么

写完这套提示词包之后，我最大的感受不是“AI 更强了”，而是项目边界终于清晰了。

以前的问题是，AI 每次都在“猜”这个项目到底想长成什么样；现在的方式则是，我先把不变量和输入结构定义清楚，再让它在约束里生成。这样做的好处很直接：

  • 项目骨架更稳定，不容易在技术栈和目录结构上跑偏
  • 接口、返回结构和权限模型更一致，前后端对齐成本更低
  • 安全边界更明确，不会在示例项目里意外塞入真实密钥
  • 验证链路更清楚，失败时能快速知道是哪一段出了问题

本质上，这并不是“更会写 Prompt”了，而是把原本隐性的工程约束，显式变成了可执行规范。

结语

我越来越觉得，所谓好的 vibe coding，并不是把所有事情都交给 AI 临场发挥，而是先把不会变的东西固化下来，再把真正该变化的部分交给生成。

对 HRMS 这种后台模板项目来说，ScaffoldSpec 负责收敛输入，三段式 Prompt 负责拆分流程，验证步骤负责守住下限。这样一来，AI 生成出来的就不再只是“一次性的代码”，而是一套可以持续复用、持续演化的脚手架。

如果以后我要继续扩展组织架构、审批流、考勤、薪资这些模块，我大概率也还会沿用这套思路：先抽取不变量，再定义输入规范，最后让生成过程有明确边界。
